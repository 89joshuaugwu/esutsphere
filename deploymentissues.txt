It looks like ESUTSPHERE is hitting a snag with react-pdf during the Vercel build. This specific error happens because newer versions of react-pdf changed their internal folder structure, or Turbopack is having trouble resolving the deep path to the CSS files.

Since you're using Next.js 16 (Turbopack), the pathing is more strict. Here is how to fix it:

1. Update the CSS Import Paths
In recent versions of react-pdf (v7 and v9+), the esm folder structure was simplified. Try updating lines 4 and 5 in ./app/(app)/library/[docId]/page.tsx to this:

TypeScript
// Change this:
import "react-pdf/dist/esm/Page/AnnotationLayer.css";
import "react-pdf/dist/esm/Page/TextLayer.css";

// To this:
import "react-pdf/dist/Page/AnnotationLayer.css";
import "react-pdf/dist/Page/TextLayer.css";
2. Check your react-pdf Version
If you recently updated the package, the path might have changed to:
import 'react-pdf/dist/Page/TextLayer.css'; (removing the /esm/ part).

Check your package.json. If you are on version 9.x, the /esm/ path definitely won't work.

3. Handle the Canvas/Worker Issue
Since this is a Next.js project, react-pdf often struggles with the canvas dependency during the build process on Vercel.

To ensure the build succeeds, add this to your next.config.mjs (or .js):

JavaScript
/** @type {import('next').NextConfig} */
const nextConfig = {
  webpack: (config) => {
    config.resolve.alias.canvas = false;
    return config;
  },
};

export default nextConfig;
4. If using Turbopack (the --turbo flag)
Your log shows Turbopack is active. Turbopack sometimes has issues resolving specific CSS imports from within node_modules. If the path change in Step 1 doesn't work:

Try running the build locally using npm run build.

If it fails locally too, you know it's a path issue.

If it works locally but fails on Vercel, try disabling Turbo in your package.json build script temporarily to see if it's a Turbopack-specific bug:

Change "build": "next build" (remove --turbo if you added it manually).

Why this happens
In TypeScript, when you use a "functional update" like setLikes(prev => ...), if the initial state wasn't clearly defined as a number, TypeScript gets worried that prev could be undefined or null. By adding : number, you are giving it the green light to proceed.
